# Nexa WebApp v1.4.0 - Bounded Context Frontend Architecture

## Overview

This release introduces the new modular frontend architecture for Nexa WebApp. The application was reorganized from feature-level root modules into the official Nexa bounded context structure, aligning the frontend with the platform architecture, backend roadmap, and academic DDD documentation.

## Main Changes

### Frontend Architecture

- Reorganized the source code around Nexa's official contexts:
  - Catalog Management
  - Sales
  - Warehouse
  - Logistics
  - Invoicing
  - IAM
  - Shared
- Preserved `app` as the application shell for routing, layouts, and composition.
- Standardized each context around application, domain, infrastructure, and presentation.
- Removed the previous root-level feature module sprawl from the active architecture.

### Catalog Management

- Moved product catalog and promotions capabilities under `catalog-management`.
- Prepared catalog infrastructure for future backend integration.

### Sales

- Moved purchase orders, purchase requests, clients, buyer portal, and commercial validation under `sales`.
- Preserved buyer-facing workflows as presentation composition inside Sales.

### Warehouse

- Moved inventory control capabilities under `warehouse`.
- Prepared inventory UI and stores for future stock endpoint integration.

### Logistics

- Moved dispatch orders, delivery tracking, customer portals, and cold-chain monitoring under `logistics`.

### Invoicing

- Moved business documents and subscriptions under `invoicing`.

### Shared

- Consolidated shared analytics, communications, utilities, layout components, HTTP infrastructure, and reusable UI components.

### Routing

- Consolidated routes per bounded context.
- Updated global router composition to load context routes instead of scattered feature routes.

### Mock API Transition

- Preserved mock data infrastructure as a temporary development fallback.
- Prepared the codebase for progressive migration from mock API to real backend endpoints.

### Documentation

- Added or updated frontend architecture documentation.
- Documented the mapping from previous feature modules to official bounded contexts.
- Updated validation evidence and release documentation.

## Validation Checklist

- `npm install`
- `npm run build`
- Context routes load correctly.
- No legacy root feature imports remain for migrated bounded contexts.
- Mock fallback remains available for non-migrated modules.
